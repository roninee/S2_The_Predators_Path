## 问题

### Question Marker

对 LLM 来说，显式结构 > 数字序号

- **没有任何公开证据**表明 `#` 在模型内部被赋予更高“权重（weight）”
- 真正起作用的是：
     **结构清晰度（structural clarity） + 模型训练中高频模式（training priors）**

> `#` 之所以“好用”，是因为它**高度结构化、低歧义、在 Markdown / prompt 中被大量使用**，而不是“权重更高”。

 

**为什么有人说**“绝对不要用 `---`”？

**原因不是模型，而是使用场景错误（context misuse）**：

| 场景                     | 是否适合   |
| ------------------------ | ---------- |
| 普通问答 / 思考型 prompt | ✅ 非常适合 |
| Markdown 文档            | ✅ 标准用法 |
| YAML / 配置文件          | ❌ 冲突     |
| 代码块内部               | ❌ 易误解析 |

**所以结论是**：

> **在自然语言提问中，`---` 是高分结束符；不是禁忌。**




| 排名 | 格式         | 示例         | 综合评分 |
| ---- | ------------ | ------------ | -------- |
| 1    | **`#`**      | `# 问题一`   | **9.5**  |
| 2    | `##` / `###` | `## 子问题`  | **9.2**  |
| 3    | `Q:`         | `Q: 为什么…` | **9.0**  |
| 4    | `-` 列表     | `- 问题一`   | 8.7      |

| 9    | `1)`         | `1) 为什么`  | 7.5      |
| 10   | **`1.`**     | `1. 为什么`  | **7.0**  |


### Question Terminator



结束符的目标不是“好看”，而是：
 **明确告诉模型：这里一个语义单元结束了（boundary signaling）**



| 排名 | 结束符             | 示例          | 综合评分 |
| ---- | ------------------ | ------------- | -------- |
| 1    | 空行（blank line） | `问题\n\n`    | **9.8**  |
| 2    | `---`              | 问题`---`     | **9.3**  |
| 3    | `### END`          | `### END`     | 9.1      |
| 4    | `【END】`          | `【END】`     | 8.9      |
| 5    | `;;`               | `;;`          | 8.6      |
| 6    | `<<< >>>`          | `<<< END >>>` | 8.4      |
| 7    | `===`              | `===`         | 8.2      |
| 8    | `//`               | `// end`      | 7.9      |
| 9    | `.` 单行           | `.`           | 7.5      |
| 10   | `;`                | `;`           | **7.2**  |





#### 六、最终建议（Actionable）

1. **问题标识**：优先用 `#` / `##`（结构 > 序号）
2. **问题结束**：
    - 单问题：空行
    - 多问题块：`---`
3. **彻底放弃**：`1. 2. 3.` 这类“人类友好、模型一般”的格式
4. **你现在的直觉是对的**：
    - 这是**结构工程（prompt structuring）**问题，不是符号迷信





## 二、引用包裹符（Quotation Delimiters）TOP 10

**评分维度（0–10）**

- 边界清晰度 boundary clarity
- 歧义风险 ambiguity risk
- 中英兼容 CN/EN compatibility
- LLM 保真度 fidelity（不被改写）
- 复杂文本承载能力（长段、多行）

## 三、为什么 ``` 是第一名（不是因为“代码”）

关键不是“代码语义”，而是三点：

1. **硬边界（hard boundary）**
    - 明确告诉模型：这是一个**独立语义块（semantic block）**
2. **训练先验（training prior）极强**
    - 大量 prompt / 文档 / QA 使用 fenced block
3. **承载力最高**
    - 长文本、多段落、符号、混中英都不变形

> 在 LLM 眼里，``` 的本质是 **“不要碰这里”** 信号。

------

## 四、各类符号的典型适用场景

### ✅ 强烈推荐

- **` `**
    - 法律条文、论文段落、聊天记录、原话
- **""" """**
    - 英文长句、逻辑段落（Python / docstring 先验）

### ⚠️ 可用但有限

- **" "** / **' '**
    - 只适合**一句话以内**
    - 多句极易被模型“吸收改写”

### ❌ 不建议作为主引用

- `()` `[]`
    - 常被模型当成**补充说明（aside）**
- `>`
    - 容易被理解为“需要总结/改写”的引用

## 七、最终结论（再压缩）

- **最佳引用包裹符**：```（fenced block）
- **一句话引用**：" "
- **中文审美但不追求极致保真**：【】
- **避免迷信**：不是符号神力，而是**边界信号强弱**

**证据强度（evidence strength）：中等偏强**
 （一致性来自主流 prompt engineering 实践 + 大量行为观察，非参数级公开）

你的疑问是合理的：

> *“主流模型以英文为主，为什么一个中文符号反而这么好用？”*

**关键反转点**：
 LLM 不是按“语言民族属性”理解符号，而是按 **统计角色（statistical role）+ 语义位置（semantic position）**。





## 、`【 】`为什么有效（核心机制）

### 1️⃣ 它是**语义型括号**，不是语法型括号

- `()` `[]` 在英文中主要承担：
    - 补充说明 aside
    - 数学 / 编程语法
- `【 】` 在真实语料中主要承担：
    - **标题（title）**
    - **标签（label）**
    - **引用头（quote header）**
    - **结构边界（section marker）**

👉 对模型来说：
 `【 】` ≈ “这是一个**人为标注的语义块**”





### 3️⃣ 边界冲突极低（这是关键优势）

对比一下：

| 符号    | 常见冲突                   |
| ------- | -------------------------- |
| `()`    | 数学、函数、补充语         |
| `[]`    | 数组、正则、脚注           |
| `" "`   | 引号嵌套、对话             |
| `【 】` | **几乎只用于人为结构标注** |

**低歧义 = 高服从度**





对“完整块、不和外界绑定”这一目标：

- **``` 是最强硬边界**
- `【 】` 是**高质量软边界**
